我不太喜欢公开不成熟的东西，因此这里只是一堆我的碎碎念，简单说一下swift的大致思路，有同仁志士看见了也可以试着做一下：

```
wasm(算 + 著录)                     原生 #bibliography(交付)            薄 Typst 粘合层
─────────────────                  ──────────────────────            ────────────────
• 解析(biblatex crate)              • 只列被引 ✓                       • AY 正文标注(注1)
• 每条成品文本(全部 A 类特性)        • 按注入键排序 ✓(刚证)             • back-ref/脚注引用
• 拼音/笔画排序键 ←auto-pinyin/bihua  • verbatim 透传 ✓                  • 单次 cite 覆盖
• 语种判定 ←glotter                 • 版式属性 ✓                    • sort-keys 手动序
（一个typst包，也是rust crate的封装）                 
• 2020a/b 消歧后缀(注2)            • LSP 补全(它就是 std.bib)✓
• /OL、PID、双语 related…
        │ 哨兵                              ▲ bytes(yaml) + 注入排序键/消歧后缀
        └──────────────────────────────────┘   show regex 还原富文本
```

我们需要写多套 `.csl` 给原生的bibliography去使用，以完美配合我们wasm的输出。

关于运行效率，可以这样朴素地假设一下：

||原生 gb-7714 CSL|swift|
|-|-|-|
|解析 `.bib`|1 次(hayagriva)|2 次(wasm*1 + hayagriva 读 `YAML` *1)|
|wasm 调用|无|1 次(+字节序列化)|
|CSL 渲染每条|重(CSL 条件/姓名宏/排序/消歧)|$\approx 0$|
|show regex后处理一下|无|1 次线性扫描 $O(n)$|

可以发现：swift 尽管多了二次解析一次 wasm的常数开销，但**省下了国标 CSL 那个最重的逐条样式计算**，两者对冲哩。加之，参考文献处理在整篇编译里本就只占极小比例，矧Typst 对插件调用和 `bibliography` 都有 memoize，增量重编一样快。予由是得一朴素结论：百条文献量级，差别在5%以内（真是纯朴素瞎说的），绝对感知不到。 只有上千条时二次解析才稍微显现，但体感上与原生差距理应仍是毫秒级。

再回顾一下整体架构，初版开发/后期维护成本包括：
- 多套并行`CSL`；
- rust src（把omni-gb7714的逻辑用rust重写，llm应该可以快速地处理好）
  - 尽量不动上游，这样可以降低维护成本
  - 如果自己解析`.bib`的话维护成本有点高，fork biblatex的话又是一笔维护成本，还得研究研究......
- typst层的后处理代码

检索了一下有关项目，应该是个novel！，但底层有大量作业可抄的思路，好耶😆，此仓的最终开发目的就是在**速度与原生完全追平**的情况下实现**远超内建国标`csl`功能**的一套方案，注意是针对国标的，不是泛化的。泛化的前提是得有海量的样式资源可用，我觉得只有三条或者说两条路可走：1.还是加载`csl`，但`csl`的功能还是有限毕竟，想让速度表现更好，唯一的根治方法还是直接给hayagriva开pr，完全没有重复造轮子的必要 2.bibtex的`.bst`样式包，这需要一个能直接加载这套逆波兰语言的rust实现，也许是合理的。至于加载biblatex的样式包，我目前觉得不太现实，不是说不能做，而是有点太复杂了。怎么做捏？基于rust的latex！目前倒是有，尽管问题多多，但真不是说不能实现，再说一遍：这项目复杂度嗖地就上去了啊 =.=。至于其他格式的参考文献样式要么太过小众了（不海量！），要么私有/半私有，没啥做的价值。
